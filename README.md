# Sistema TAZZIN

> **Menos planilhas. Mais controle.**

TAZZIN is not a closed product — it is a **showcase of independent modules**.
Each one (Sales, Customer Relationship, Opportunity Radar, Production,
Machining, Operators, Scrap, Time Adjustments, Inventory, Purchasing, Finance,
People, Compensation, Projects, Maintenance, Quality, Administration) solves a
real operational problem, runs on its own, and can be adopted individually and
fully customized to a company's reality. The home page is a **vitrine** that
presents them as a catalog; an optional Executive dashboard shows what they can
do when combined. Adopt one tool — the one that hurts today — and grow one
module at a time.

The dataset in this repository is **synthetic** — generated from scratch, so
the whole system can be explored end to end without exposing any real
company's data. See [`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md) for how it
is organized and why, and [`docs/marca/`](docs/marca) for the brand manual
the interface follows.

## Highlights

- **Layered architecture**: Interface (Streamlit) → Services → Domain rules
  → Repositories → Database, with a strict one-way dependency direction.
- **13 business modules** across 15 pages, each independently testable,
  sharing only a generic repository base and a handful of cross-module id
  references (never ORM relationships across module boundaries — see
  [Architecture § Module boundaries](docs/ARCHITECTURE.md#module-boundaries)).
- **38 tables**, ~46,000 synthetic rows generated by [`scripts/generate_synthetic_data.py`](scripts/generate_synthetic_data.py),
  spanning January 2023 to today.
- **244 automated tests** (domain rules, repositories, services) running
  against an isolated in-memory SQLite database — no shared state between
  tests.
- **Observability built in**: structured JSON logging, a health-check
  module (database + filesystem probes), and an in-process metrics
  registry (call counts, latency, error counts per operation), all
  surfaced on the Administration page.

## Quick start

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

python scripts/generate_synthetic_data.py --reset   # populate data/tazzin.db
streamlit run app/main.py                           # open http://localhost:8501
```

See [`docs/HOW_TO_RUN.md`](docs/HOW_TO_RUN.md) for details, configuration
via environment variables, and how to point at PostgreSQL instead of the
bundled SQLite file.

## Project layout

```
app/
  core/            settings, logging, exceptions, health check, metrics
  database/        SQLAlchemy engine/session + one model module per business module
  repositories/    generic CRUD base + one repository per module
  domain/          pure business-rule functions (no I/O), one module per business module
  services/        orchestrates repositories + domain rules; the only layer the UI talks to
  pages/           Streamlit pages, one per module
  Início.py        Vitrine — catálogo de módulos independentes (home page)
  main.py          Streamlit entrypoint — registers pages via st.navigation
scripts/           init_db.py, generate_synthetic_data.py, run_health_check.py
tests/             mirrors app/ — test_domain/, test_repositories/, test_services/
docs/              architecture, module reference, how-to-run, contributing
```

See [`docs/MODULES.md`](docs/MODULES.md) for what each business module
covers and [`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md) for the layering
rules and the reasoning behind them. For a complete technical reference —
architecture diagrams, domain model, data flow, improvement points and
roadmap — see [`docs/DOCUMENTACAO_TECNICA.md`](docs/DOCUMENTACAO_TECNICA.md)
(pt-BR).

## Running tests

```bash
pytest                    # 244 tests, ~3s, in-memory SQLite
ruff check app tests scripts
```

## Generating a fresh dataset

```bash
python scripts/generate_synthetic_data.py --reset --seed 42
```

See [`docs/DATA_GENERATION.md`](docs/DATA_GENERATION.md) for the
generation order, volume per table, and how upstream activity (sales,
purchasing, production) drives downstream activity (stock movements,
invoices) so the dataset reads as one coherent company rather than
independently randomized tables.
